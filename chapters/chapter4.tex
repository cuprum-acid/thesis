\chapter{Implementation}\label{chap:impl}

This chapter describes the implementation of the two systems introduced in Chapter~\ref{chap:met}. Both systems were developed in Go to ensure consistency in comparison, but differ significantly in architecture, execution model, and state management. The goal is to provide detailed insight into implementation decisions, internal logic, and system behavior.

\section{Implementation Overview}

Both systems implement identical business logic for billing and subscription management:
\begin{itemize}
    \item \textbf{BillingPlan}: Pricing configuration with price, currency, billing period
    \item \textbf{Subscription}: User subscription linked to a plan with lifecycle management
    \item \textbf{Billing cycle}: Automatic recurring billing with configurable intervals
    \item \textbf{Expiration handling}: State transitions when billing conditions are not met
    \item \textbf{Final charge}: Cleanup logic on subscription deletion
\end{itemize}

The implementations differ in:
\begin{itemize}
    \item \textbf{State persistence}: PostgreSQL vs etcd via CRD
    \item \textbf{Execution model}: Request-response vs reconcile loop
    \item \textbf{Validation}: Runtime validation vs admission control
    \item \textbf{Observability}: HTTP/DB tracing vs Events/Conditions/Reconciliation metrics
\end{itemize}

\subsection{Controller Implementation}

The core business logic resides in \texttt{SubscriptionReconciler} (\texttt{internal/controller/subscription\_controller.go}).

\subsubsection{Reconcile Method}

The \texttt{Reconcile} method implements the 6-step workflow:

\begin{lstlisting}[language=Go]
func (r *SubscriptionReconciler) Reconcile(ctx context.Context, 
    req ctrl.Request) (ctrl.Result, error) {
    
    // Step 1: Fetch Subscription
    var sub Subscription
    if err := r.Get(ctx, req.NamespacedName, &sub); err != nil {
        return ctrl.Result{}, client.IgnoreNotFound(err)
    }
    
    // Step 2: Handle deletion
    if !sub.DeletionTimestamp.IsZero() {
        return r.handleDeletion(ctx, &sub)
    }
    
    // Step 3: Add finalizer
    if !controllerutil.ContainsFinalizer(&sub, FinalizerName) {
        controllerutil.AddFinalizer(&sub, FinalizerName)
        return r.Update(ctx, &sub)
    }
    
    // Step 4: Validate BillingPlan
    plan, err := r.getPlan(ctx, sub.Spec.PlanRef)
    if err != nil {
        return r.setErrorCondition(ctx, &sub, err)
    }
    
    // Step 5: Initialize if new
    if sub.Status.State == "" {
        return r.activateSubscription(ctx, &sub, plan)
    }
    
    // Step 6: Process billing cycle
    if sub.Status.State == StateActive && time.Now().After(sub.Status.NextBilling.Time) {
        return r.processPayment(ctx, &sub, plan)
    }
    
    return ctrl.Result{RequeueAfter: time.Duration(plan.Spec.RequeueIntervalSeconds) * time.Second}, nil
}
\end{lstlisting}

\subsubsection{Watches and Event Mapping}

The controller watches both Subscriptions and BillingPlans:

\begin{lstlisting}[language=Go]
func (r *SubscriptionReconciler) SetupWithManager(mgr ctrl.Manager) error {
    return ctrl.NewControllerManagedBy(mgr).
        For(&billingv1alpha1.Subscription{}).
        Watches(
            &billingv1alpha1.BillingPlan{},
            handler.EnqueueRequestsFromMapFunc(r.mapPlanToSubscriptions),
        ).
        Complete(r)
}

func (r *SubscriptionReconciler) mapPlanToSubscriptions(ctx context.Context, 
    obj client.Object) []reconcile.Request {
    
    plan := obj.(*billingv1alpha1.BillingPlan)
    var subs SubscriptionList
    r.List(ctx, &subs, client.MatchingFields{"spec.planRef": plan.Name})
    
    var requests []reconcile.Request
    for _, sub := range subs.Items {
        requests = append(requests, reconcile.Request{
            NamespacedName: types.NamespacedName{
                Namespace: sub.Namespace,
                Name:      sub.Name,
            },
        })
    }
    return requests
}
\end{lstlisting}

func (h *SubscriptionHandler) CreateSubscription(w http.ResponseWriter, r *http.Request) {
    var req CreateSubscriptionRequest
    if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
        http.Error(w, err.Error(), http.StatusBadRequest)
        return
    }
    
    // Validate
    if err := validator.Validate(req); err != nil {

        PlanRef: req.PlanRef,
        State:   "Active",

        return
    }

    var subs []*models.Subscription
    err := r.db.WithContext(ctx).
        Where("state = ? AND next_billing < ?", "Active", now).

\subsection{Background Worker}

The expiration checker runs every 30 seconds:

\begin{lstlisting}[language=Go]
// workers/workers.go
func StartExpirationChecker(ctx context.Context, repo SubscriptionRepository, interval time.Duration) {
    ticker := time.NewTicker(interval)
    defer ticker.Stop()
    
    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            expired, err := repo.FindExpired(ctx, time.Now())
            if err != nil {
                log.Error("Failed to find expired", "error", err)
                continue
            }
            
            for _, sub := range expired {
                sub.State = "Expired"
                if err := repo.Update(ctx, sub); err != nil {
                    log.Error("Failed to update", "id", sub.ID, "error", err)
                }
            }
            log.Info("Expiration check completed", "expired", len(expired))
        }
    }
}
\end{lstlisting}

\begin{itemize}

\section{Implementation Challenges}

\subsection{Kubernetes System Challenges}

\begin{itemize}
    \item \textbf{Steep learning curve}: Kubebuilder, controller-runtime, CRD schema
    \item \textbf{Debugging complexity}: Asynchronous reconcile loop, eventual consistency
    \item \textbf{Schema evolution}: CRD versioning and migration strategies
    \item \textbf{Testing overhead}: envtest setup, Kind cluster management
\end{itemize}

\subsection{Traditional System Challenges}

\begin{itemize}
    \item \textbf{Manual lifecycle management}: No built-in finalizer equivalent
    \item \textbf{Polling overhead}: Background worker continuously queries database
    \item \textbf{Scaling complexity}: Manual horizontal scaling with load balancer
    \item \textbf{State recovery}: No automatic reconstruction from declarative state
\end{itemize}

\section{Summary}

This chapter presented detailed implementation of both systems. The Kubernetes-based system leverages CRDs, Operators, and the reconcile pattern to implement declarative business logic. The traditional system uses REST APIs, PostgreSQL, and background workers for imperative logic. Key differences include state management (etcd vs PostgreSQL), execution model (event-driven vs request-response), and error handling (eventual consistency vs explicit errors). These implementation details form the basis for the experimental evaluation in Chapter~\ref{chap:eval}.
