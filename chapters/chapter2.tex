\chapter{Literature Review}\label{chap:lr}

\section{Introduction}

This chapter provides a comprehensive review of the literature and technologies relevant to exploring Kubernetes as a platform for business logic implementation. The chapter is organized into several sections covering Kubernetes architecture and extensibility, Custom Resource Definitions and Operators, related approaches to extending Kubernetes, observability in cloud-native systems, and performance considerations.

\section{Kubernetes Architecture and Core Concepts}

\subsection{Container Orchestration and Kubernetes}

Kubernetes, initially developed by Google and open-sourced in 2014, has become the leading platform for container orchestration~\cite{kubernetes, luksa2017}. The system builds upon years of experience running production workloads at Google, combined with best-of-breed ideas from the open-source community~\cite{cncf, hightower2017}. Kubernetes automates deployment, scaling, and management of containerized applications, providing a platform-agnostic abstraction layer over underlying infrastructure.

The architecture of Kubernetes follows a control plane pattern, where a centralized control plane manages the desired state of the cluster, and worker nodes execute containerized workloads~\cite{k8s_architecture, luksa2017}. The control plane consists of several key components: the API server, which exposes the Kubernetes API and serves as the entry point for all operations; etcd, a distributed key-value store that maintains cluster state; the scheduler, which assigns pods to nodes; and the controller manager, which runs controllers to reconcile actual state with desired state~\cite{ibryam2020}.

\subsection{Declarative API Model}

A distinguishing feature of Kubernetes is its declarative API model~\cite{kubernetes_declarative_automation}. Rather than imperatively instructing the system on how to perform operations, users declare the desired state of resources, and Kubernetes automatically reconciles the actual state to match. This declarative approach enables powerful automation capabilities and forms the foundation for extending Kubernetes functionality~\cite{hausenblas2019}.

The Kubernetes API is organized into API groups and versions, providing a structured approach to API evolution and extensibility~\cite{k8s_api}. Core resources like Pods, Services, and Deployments are part of the built-in API, while custom resources can be added through extension mechanisms. This design has been validated through extensive production use and is thoroughly documented in standard references~\cite{luksa2017, hightower2017}.

\subsection{Controller Pattern and Reconciliation Loop}

Kubernetes extensively uses the controller pattern for managing resources~\cite{controller_pattern, ibryam2020}. A controller watches for changes to resources of interest and takes actions to reconcile the actual state with the desired state specified in the resource definition. This reconciliation loop runs continuously, enabling self-healing and automated management capabilities~\cite{dobies2020}. Figure~\ref{fig:controller-pattern} sketches the canonical control flow: a watch on the API server delivers events into the controller's work-queue; the reconciler dequeues an item, reads current spec and status from the API server, compares them to the desired state, makes any necessary changes (which themselves become API-server writes), and either schedules a future reconciliation through \texttt{RequeueAfter} or returns and waits for the next event.

Controllers operate independently and asynchronously, communicating only through the Kubernetes API server. This architectural pattern enables high scalability and resilience, as controllers can be distributed across multiple instances and automatically recover from failures~\cite{hausenblas2019}. The controller pattern has become a fundamental building block for cloud-native applications and is extensively covered in Kubernetes design patterns literature~\cite{ibryam2020}.

\section{Custom Resource Definitions}

\subsection{Extending the Kubernetes API}

Kubernetes provides mechanisms for extending its API, with Custom Resource Definitions (CRDs) being the primary approach~\cite{extending_k8s_api, hausenblas2019}. CRDs represent a declarative approach to defining new resource types without requiring custom API server code, making them accessible to operators and developers~\cite{luksa2017}.

A CRD defines a new resource type, including its API group, version, kind, and schema, making it a first-class citizen in the Kubernetes API~\cite{crd_kubernetes, hausenblas2019}. This approach enables extending Kubernetes to manage domain-specific resources while maintaining consistency with the platform's declarative model.

\subsection{CRD Features and Capabilities}

Modern CRDs support advanced features including schema validation using OpenAPI v3, multiple versions with conversion webhooks, status subresources for privilege separation, custom columns in kubectl output, and structural schemas for efficient storage and processing~\cite{crd_features, hausenblas2019}. These capabilities enable CRDs to provide rich, production-ready APIs comparable to built-in Kubernetes resources.

CRDs enable developers to model custom workflows, advanced configurations, and support for domain-specific requirements that go beyond Kubernetes' built-in resources. The flexibility of CRDs has made them a cornerstone of cloud-native extensibility, with hundreds of widely-used CRDs available in the ecosystem~\cite{kubernetes_multivocal_review}.

\subsection{Performance and Scalability Considerations}

The performance characteristics of CRDs depend heavily on the underlying etcd datastore, which stores all cluster state. Research on etcd performance shows that at scale, write latency can become a bottleneck, particularly for clusters with many CRD instances~\cite{etcd_impact}. Organizations deploying CRD-intensive applications must carefully consider etcd capacity planning and implement appropriate monitoring.

\section{The Operator Pattern}

\subsection{Operator Fundamentals}

The Operator pattern represents a methodology for packaging, deploying, and managing Kubernetes applications by extending the Kubernetes API~\cite{operator_pattern, dobies2020}. Operators combine CRDs with custom controllers to encode operational knowledge about managing complex applications, effectively acting as automated site reliability engineers~\cite{openshift_operators_lifecycle}.

Operators capture the aim of human operators who manage services—they have deep knowledge of how systems should behave, how to deploy them, and how to react to problems. By codifying this operational knowledge, Operators enable applications to be self-managing, automatically handling tasks like upgrades, backups, scaling, and failure recovery~\cite{dobies2020, ibryam2020}.

\subsection{Operator Development Frameworks}

Although the Operator pattern is framework-agnostic in principle, in practice every non-trivial controller is built on a higher-level scaffold that hides the watch-list-cache-reconcile boilerplate. Four frameworks dominate the space and differ along several axes that influence the choice for business-logic controllers.

Kubebuilder~\cite{kubebuilder} is the SIG-API-Machinery-maintained framework that this thesis uses. It generates a project skeleton with a single language (Go), wires up \texttt{controller-runtime} and \texttt{controller-tools}, and produces both Go types and OpenAPI CRD manifests from kubebuilder markers attached to struct fields. Its dominance in the operator ecosystem follows from its proximity to the upstream Kubernetes project: the same primitives underlying \texttt{kube-controller-manager} are reused, and updates track Kubernetes releases closely~\cite{hausenblas2019}.

The Red Hat Operator SDK extends Kubebuilder with additional code generators (Ansible-driven and Helm-driven operator scaffolds) and tighter integration with the Operator Lifecycle Manager~\cite{openshift_operators_lifecycle}. For Go-based operators the two frameworks are largely interchangeable; the Operator SDK is preferable when the operator is intended to ship through OperatorHub, where OLM-aware manifests are mandatory.

Metacontroller (originally Google, now community-maintained) inverts the control structure: the developer writes a webhook returning desired child objects given the parent's state, and Metacontroller is the controller that calls the webhook. This is attractive for declarative compositions where the operator's job is to render N child resources from M parents, but the lack of language constraints comes at the cost of an indirection that complicates debugging.

KUDO and KubeOps target operator authors outside the Go ecosystem (a YAML DSL and .NET respectively) and are accordingly less common for general-purpose business logic. They illustrate that the operator pattern is portable across language ecosystems even though the reference implementation is Go.

For the business-logic use case explored in this thesis, Kubebuilder is the only framework that combines (i)~native Go performance, (ii)~direct access to the controller-runtime metrics registry needed for instrumentation, and (iii)~a mature testing harness (\texttt{envtest}); the comparison in subsequent chapters is therefore conducted against Kubebuilder-shaped operators. Studies analysing operator bugs across hundreds of Kubernetes operators identify state management, error handling, and race conditions as the dominant failure modes~\cite{operator_bugs}, regardless of framework choice --- a finding that motivates the explicit attention this thesis pays to the Conditions API and finalizer ordering in Chapters~\ref{chap:met} and~\ref{chap:impl}.

\subsection{Operator Use Cases and Applications}

Operators have been successfully applied to numerous domains including database management, monitoring systems, service meshes, and machine learning platforms~\cite{dobies2020, openshift_operators_lifecycle}. For infrastructure provisioning, Crossplane exemplifies using Operators to manage cloud resources declaratively through Kubernetes~\cite{crossplane}. This approach enables treating infrastructure as Kubernetes resources, managed through the same workflows as application deployments.

\section{Related Systems and Their Boundaries}

The CRD-and-controller pattern is used in three distinct settings: infrastructure provisioning, network and policy management, and (the focus of this thesis) business workflow management. The first two are mature and well-studied; the third is the gap this thesis addresses. Setting the scope explicitly clarifies what is and is not new about the proposed work.

\subsection{Infrastructure as Code, GitOps, and Crossplane}

Infrastructure as Code (IaC) and Kubernetes operators share the language of declarative desired state but disagree on the execution model. Tools like Terraform and Pulumi compile a desired configuration into a one-shot apply step: the agent runs once, converges, and exits. The control loop is external, lives in the developer's CI pipeline, and only fires on explicit invocation~\cite{iac_devops, kubernetes_declarative_automation}. A Kubernetes operator, by contrast, runs continuously and reconciles every change event; there is no apply boundary, and drift between declared and actual state is detected within seconds of its appearance.

GitOps shifts that boundary back inside the cluster: an in-cluster agent (Argo~CD, Flux) holds a working copy of a Git repository and continuously reconciles the cluster toward whatever the current commit declares~\cite{gitops, argocd}. The agent itself is a controller; what was an external IaC tool becomes another reconciliation loop inside the same cluster. GitOps and the operator pattern therefore sit on a spectrum, with the agent treating Kubernetes resources as the desired state and the operator treating CRDs as the desired state.

Crossplane is the clearest example of operators in the IaC role: it exposes cloud-provider primitives (S3 buckets, RDS instances, IAM roles) as CRDs, and a controller per provider drives the actual cloud resources toward the declared state~\cite{crossplane}. The operator pattern thus has an established use in infrastructure provisioning. What distinguishes the present thesis is the \emph{scope of the reconciled object}: Crossplane reconciles cloud infrastructure; this work reconciles a business entity whose lifecycle (subscriptions, billing cycles, refunds) has no infrastructure component at all.

\subsection{Service Meshes as Heavy CRD Consumers}

Service meshes such as Istio illustrate that CRDs scale to operationally complex domains: a single Istio installation registers dozens of CRDs (\texttt{VirtualService}, \texttt{DestinationRule}, \texttt{AuthorizationPolicy}, \texttt{PeerAuthentication}, and others)~\cite{istio, nist_service_mesh_security}. Each is a piece of policy that the mesh control plane reconciles into proxy configuration. Operationally this is impressive --- but the scope is still ``cluster-wide infrastructure policy''. The service mesh case is instructive in two ways: it demonstrates that the CRD pattern can sustain complex hierarchies of declarative configuration, and it sets a precedent for treating the policy layer as a first-class API surface. Both observations transfer to the business-logic case, but service meshes do not themselves constitute business-logic operators.

The category boundary is not merely cosmetic. Application-lifecycle operators reconcile Kubernetes resources (Pods, Services, PersistentVolumeClaims) on behalf of a stateful workload they package, and their cleanup hooks usually orchestrate pod-level operations. Business-workflow operators reconcile resources whose canonical representation \emph{is} the CRD: there is no underlying Pod or external system being managed; the resource is the work. This shift in reference frame is what the present thesis investigates empirically and what motivates the comparison against a traditional REST stack rather than against another operator.

\section{Observability in Cloud-Native Systems}

\subsection{Monitoring and Metrics Collection}

Observability in distributed systems encompasses metrics, logs, and traces. Prometheus has emerged as the de facto standard for metrics collection in Kubernetes environments~\cite{prometheus, prometheus_cloud_native_monitoring}. Its pull-based model, powerful query language (PromQL), and native Kubernetes service discovery make it well-suited for cloud-native monitoring~\cite{hpc_monitoring_kubernetes_prometheus}.

For CRD-based applications, proper instrumentation of controllers and business logic is essential for understanding system behavior. Controllers should expose metrics about reconciliation loop performance, error rates, and business-specific indicators.

\subsection{Distributed Tracing}

OpenTelemetry represents the industry standard for distributed tracing and observability instrumentation~\cite{opentelemetry}. Distributed tracing becomes particularly important when business logic spans multiple reconciliation loops and controllers, as it enables tracking the cascading changes of objects through the Kubernetes control plane.

The integration of observability tools with Kubebuilder-based Operators is well-established, with frameworks providing automatic instrumentation of common controller metrics. However, domain-specific business metrics require custom implementation within the reconciliation logic.

\section{Performance Studies and Challenges}

\subsection{Kubernetes Performance Characteristics}

Studies have examined Kubernetes performance under various conditions. Research on etcd deployment impact demonstrates that control plane performance and application performance both depend critically on etcd performance characteristics~\cite{etcd_impact}. Write latency and throughput limitations in etcd can become bottlenecks for CRD-intensive workloads.

\subsection{Complexity and Operational Considerations}

While Kubernetes provides powerful capabilities, it introduces significant complexity and operational overhead~\cite{kubernetes, kubernetes_multivocal_review}. Managing a production Kubernetes cluster requires expertise in networking, security, storage, and distributed systems. Organizations must weigh these operational costs against the benefits of Kubernetes' automation and standardization.

For business logic implementations specifically, additional complexity arises from the need to design appropriate CRD schemas, implement robust controllers, and handle edge cases in the reconciliation logic~\cite{hausenblas2019}. The learning curve for Kubernetes development is substantial, potentially impacting development velocity.

\section{Research Gaps}

The literature surveyed above documents the operator pattern thoroughly in three categories --- infrastructure provisioning (Crossplane and provider-specific operators), application lifecycle (database and ML-platform operators), and policy/network (service meshes). The business-workflow category is essentially absent from the academic literature, and the few practitioner reports that do describe such operators do not measure them. The specific gaps this thesis addresses are:

\begin{enumerate}
    \item \textbf{Absence of empirical comparison against a non-operator baseline for equivalent business logic.} The multi-vocal review~\cite{kubernetes_multivocal_review} catalogues benefits and challenges of Kubernetes adoption but reports no controlled comparison between a CRD-based and a non-CRD-based implementation of the same business workflow; the existing performance literature focuses on etcd as a substrate~\cite{etcd_impact} rather than on the application-level cost of running business logic above it.
    \item \textbf{Lack of treatment of business-workflow operators as a distinct category.} Existing operator studies (Operator SDK literature~\cite{openshift_operators_lifecycle}, the Operator-bugs empirical study~\cite{operator_bugs}) examine operators built for infrastructure or application-lifecycle purposes; the question of whether the operator pattern is suitable for reconciling \emph{business} entities is not separated from the question of whether it is suitable for reconciling \emph{infrastructure} entities, even though the two have substantially different latency and throughput requirements.
    \item \textbf{No principled discussion of construct asymmetries between operator and REST architectures.} Comparisons that do exist (in technical blogs and conference talks) tend to align the two architectures at the feature level and underspecify the cases where they cannot be aligned --- cancellation, error reporting, audit trails. The asymmetry itself is treated as an implementation detail rather than as a finding.
    \item \textbf{Limited evidence-based guidance on when the operator pattern is justified for business logic.} The decision-framework material in practitioner literature is largely intuition-driven; the multi-vocal review notes the gap but does not fill it.
\end{enumerate}

These gaps motivate the structure of the rest of the thesis: a controlled comparison between two reference implementations of the same business workflow (Chapter~\ref{chap:impl}), measured along latency, throughput, resource utilisation, and operational dimensions (Chapter~\ref{chap:eval}), with construct asymmetries reported as findings rather than smoothed over (Chapter~\ref{chap:conclusion}).

\section{Summary}

This chapter has reviewed the relevant literature on Kubernetes architecture, extensibility mechanisms, and related technologies. Kubernetes provides a powerful platform for building cloud-native applications through its declarative API, CRDs, and Operator pattern~\cite{luksa2017, hightower2017, dobies2020}. While extensively used for infrastructure management, its application to general business logic remains understudied~\cite{kubernetes_multivocal_review}. The next chapter describes the design of systems to explore this research question.
